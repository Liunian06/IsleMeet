# AGENTS.md

## 项目背景

- 屿见 / IsleMeet 是一个 AI 陪伴类跨平台 APP。
- 屿见采用类似递归式系统的设计：用户打开的是 APP，实际承载的是一个完整的 AI-native AI 陪伴系统。
- 正式架构方向为 `C++17 + Flutter UI + SQLite`。
- 本机 Flutter 位置为 `E:\NewSoftware\flutter`。

## 协作范式

- 用户先提出需求，AI 先理解并反馈关键点、歧义和缺失信息。
- AI 先输出一份简明 PRD，写入 `docs/` 目录，供用户审核。
- 用户确认 PRD 后，AI 再进入实现。
- 实现阶段采用小步迭代，每次只推进一批明确变更。
- 由于这是跨平台 APP，所有实现都要尽量由 AI 自主编译，并在可用真机上验证。
- 遇到编译错误、平台差异、运行时异常、适配问题或真机问题时，优先修复，再继续下一轮。
- 以真机可用作为交付标准，不以“代码看起来正确”作为最终完成标准。
- 每轮迭代结束后同步更新 PRD 和实现状态，保持文档和代码一致。

## 资源约束

- 开发过程中，所有 icon 都必须由 AI 手绘 SVG 完成，不使用现成图标素材。
- 开发过程中，不使用 emoji 代替 icon。

## UI 风格

- 优先参考原生 iOS 的极简线条风格。
- 整个项目的 UI 设计应保持简约、治愈的方向。
- 留白充足，页面不要塞满，信息之间要有清晰间距，降低阅读负担。
- 视觉装饰要少，少用重阴影、厚边框、复杂渐变和花哨背景。
- 字体优先承担层级表达，主要依靠字号、字重和颜色深浅建立信息层次。
- 强调系统感，优先使用系统控件、系统动效、系统颜色和列表样式。
- 导航路径要明确，常见结构优先使用底部 Tab、返回导航、分组列表和设置页式布局。
- 页面应优先呈现内容和操作，而不是先展示设计本身。
- 交互反馈要细腻但不过度，按钮按压、切换、滚动和弹窗反馈都应轻微、顺滑、自然。
- 图标应保持简洁统一，采用线性、单色、语义清楚的表达方式。
- 圆角保持适中，不做得过软或过厚重，整体更像工具界面而不是营销页。
- 不使用玻璃磨砂质感。
- 不使用蓝紫色科技风。
- 不使用与上述风格相近的装饰性重效果。

## 执行原则

- 协作模式以“沟通 -> 执行 -> 验证 -> 修复 -> 再验证”为闭环，聊天只负责澄清目标和边界，不作为交付终点。
- AI 的主要职责是推进任务完成，而不是只做解释性回答。
- 需要文件时先读文件，需要改动时直接修改，需要验证时直接编译和运行，需要证据时直接查看输出和日志。
- 遇到问题先回到当前闭环中修复，不把本应由 AI 自己处理的步骤转回给用户。
- AI 应持续维护项目上下文、决策记录和任务状态，确保跨轮次连续推进。
- 先思考再编码。若存在多种解释，AI 先说明假设和取舍，不默认替用户做隐含决定。
- 需求不清时先停下澄清，不靠猜测推进。
- 只实现当前明确要求的内容，不增加未请求的功能或抽象。
- 优先采用最简单、最直接、可验证的方案。
- 修改已有代码时，只触及必要范围，不重构无关部分，不清理与本次变更无关的旧代码。
- 每次改动都应能直接对应到用户需求，并且能被验证。
- 对多步骤任务，AI 先给出简短计划，再按步骤执行并在每步后验证结果。
- 所有文件修改都必须使用 `apply_patch` 工具完成，不使用 PowerShell 直接写入文件。
- 在回复中展示 Windows 本地路径时，普通路径使用代码格式，例如 `E:\NewAI\IsleMeet2\README.md`。
- 在回复中提供可点击本地文件链接时，Markdown 链接目标必须使用正斜杠路径，例如 `[README.md](E:/NewAI/IsleMeet2/README.md)`，不要把 `E:\...` 反斜杠路径直接放进 Markdown 链接目标。
- 路径包含空格时，可点击 Markdown 链接目标必须使用尖括号包裹，并使用正斜杠路径。
